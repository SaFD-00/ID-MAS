# Reflexion: Fallback Skip + Logging 계획 수립

**날짜**: 2026-01-16

---

## 1. 계획 수립 과정 요약

### 작업 내용
ID-MAS train 과정에서 fallback 발생 시 해당 문제를 skip하고 failure로 로그에 기록하는 기능 계획 수립

### 진행 과정
1. **Phase 1**: Codebase 탐색 → 기존 fallback 메커니즘 이미 구현되어 있음 발견
2. **Phase 2**: 사용자 인터뷰 → "Skip + Log" 방식 확정
3. **Phase 3-4**: Spec & Architecture, Task Decomposition 작성
4. **Phase 5**: Verifier 검증 → 누락된 변경 사항 발견 → 계획 수정

---

## 2. 효과적이었던 패턴

### 2.1 병렬 탐색
- Explorer 에이전트와 로그 파일 읽기를 병렬로 실행하여 시간 절약
- 코드 구조와 실제 데이터 형식을 동시에 파악

### 2.2 사용자 인터뷰를 통한 요구사항 명확화
- 초기에 "fallback 등록"이라는 모호한 요청을 구체적인 요구사항으로 변환
- 3가지 선택지(Skip, Continue, Retry) 제시하여 명확한 결정 도출

### 2.3 Verifier를 통한 계획 검증
- 계획 작성 후 자동 검증으로 누락 사항 발견
- 특히 state.py의 여러 관련 함수들의 수정 필요성 확인

---

## 3. 개선이 필요한 패턴

### 3.1 계획 문서의 라인 번호 부정확
- 초기 탐색 시 확인한 라인 번호가 계획 문서에 잘못 기재됨
- **개선**: 코드 참조 시 상대적 위치(함수명, 근처 특징) 함께 기록

### 3.2 연관 함수 분석 누락
- create_initial_state(), load_checkpoint_from_logs() 등 연관 함수 수정 누락
- **개선**: 필드 추가 시 해당 필드를 사용/초기화하는 모든 위치 검색

### 3.3 Task Decomposition 세분화 부족
- 초기에 T1-T7로 큰 단위로만 분해
- 검증 후 T2.1-T2.4, T3.1-T3.3으로 세분화 필요
- **개선**: 한 파일 내 여러 함수 수정 시 각각을 별도 Task로 분리

---

## 4. 학습된 인사이트

### 4.1 기존 코드 구조 존중
- ID-MAS는 이미 `_failure_metadata` 패턴을 일관되게 사용
- 새 기능 추가 시 기존 패턴을 따르는 것이 유지보수에 유리

### 4.2 TypedDict 확장 시 주의점
- `total=False`인 TypedDict에 필드 추가는 안전
- 하지만 해당 필드를 사용하는 모든 함수 업데이트 필요

### 4.3 Resume 기능 고려
- checkpoint/resume 기능이 있는 시스템에서 새 필드 추가 시
- 이전 checkpoint 파일과의 호환성 고려 필요
